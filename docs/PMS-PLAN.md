# Benjamin PMS — แผนการสร้างและออกแบบระบบ

> **ประเภทระบบ:** Project Management System (PMS) แบบ Multi-tenant สำหรับธุรกิจอาคารโครงสร้างเหล็กสำเร็จรูป (PEB)
> **โมเดลผู้ใช้:** สำนักงานใหญ่ (HQ) เห็นทุกอย่าง / ดีลเลอร์ (Dealer) เห็นเฉพาะข้อมูลของตัวเอง
> **สถานะ:** แผน & การออกแบบโครงสร้าง (ยังไม่เริ่มเขียนโค้ด)

---

## 1. เป้าหมายของระบบ (Goals)

รวมงานขาย–เสนอราคา–บริหารโครงการ–รายงาน ของ Benjamin และเครือข่ายดีลเลอร์ทั่วประเทศไว้ในที่เดียว เพื่อให้:

- **HQ** มองเห็นภาพรวมทั้งเครือข่าย: ลีด ยอดขาย สถานะโครงการ ประสิทธิภาพดีลเลอร์ แบบ real-time
- **ดีลเลอร์** มีเครื่องมือทำงานครบ (CRM, ใบเสนอราคา, ติดตามโครงการ) โดยเห็นเฉพาะข้อมูลของตน
- ลดงานเอกสารซ้ำซ้อน, มาตรฐานราคาและ BOQ เป็นแบบเดียวกัน, ติดตามงานตาม 6 เฟสของ Benjamin ได้

---

## 2. ผู้ใช้และสิทธิ์ (Roles & Access)

ระบบใช้ **2 มิติ**: (ก) สังกัด (HQ หรือ Dealer ใด) + (ข) บทบาทการทำงาน

| Role | สังกัด | ขอบเขตการมองเห็น | หน้าที่หลัก |
|------|--------|------------------|-------------|
| **Super Admin** | HQ | ทั้งระบบ + ตั้งค่า | ดูแลระบบ, ผู้ใช้, master data |
| **HQ Management** | HQ | ทุกดีลเลอร์ (read ทั้งหมด) | แดชบอร์ดผู้บริหาร, อนุมัติ, รายงาน |
| **HQ Sales/Engineer** | HQ | ทุกดีลเลอร์ | ช่วยงานขาย/ออกแบบข้ามเขต, คุมมาตรฐาน BOQ |
| **Dealer Admin** | Dealer X | เฉพาะ Dealer X | จัดการผู้ใช้/ข้อมูลของดีลเลอร์ตน |
| **Dealer Sales** | Dealer X | เฉพาะลีด/ลูกค้าที่ได้รับมอบหมาย | ดูแลลีด, ทำใบเสนอราคา |
| **Dealer Project/Site** | Dealer X | เฉพาะโครงการของตน | อัปเดตหน้างาน, รูป, QC, รายงานประจำวัน |
| **Customer (เฟสหลัง)** | – | เฉพาะโครงการของตัวเอง | ดูสถานะ/เอกสารโครงการตน (read-only portal) |

**กฎความปลอดภัยหลัก (Tenant Isolation):** ทุก record ผูก `dealer_id`
- ผู้ใช้ดีลเลอร์ → query ถูกกรองด้วย `dealer_id` ของตนเสมอ (Row-Level Security)
- ผู้ใช้ HQ → bypass ตัวกรอง เห็นได้ทั้งหมด
- การ "โอนลีด/โครงการ" ข้ามดีลเลอร์ ทำได้เฉพาะ HQ หรือ Dealer Admin ต้นทาง

## 3. โมดูลหลัก 4 โมดูล

### 3.1 CRM — ลีด / ลูกค้า / ดีลเลอร์
- **กรอกลูกค้า/ลีดเอง (ช่องทางหลัก)**: ดีลเลอร์กรอกข้อมูลลูกค้าและลีดเข้าระบบเอง → ผูกเป็นของดีลเลอร์ตนทันที
- **Lead Inbox (เสริม)**: รับลีดอัตโนมัติจาก *ฟอร์มติดต่อบนเว็บ benjaminpebsteel.com*, LINE OA, โทร, walk-in เข้า pool กลาง
- **Lead Routing**: ลีดจาก pool กลาง → HQ มอบหมายให้ดีลเลอร์ตามจังหวัด/พื้นที่ (ลีดที่ดีลเลอร์กรอกเองไม่ต้องผ่านขั้นนี้)
- **Pipeline (Kanban)**: ลีดใหม่ → ติดต่อ → สำรวจ → เสนอราคา → ต่อรอง → ปิดการขาย (ชนะ/แพ้)
- **Customer 360**: ประวัติลูกค้า, กิจกรรม (call/visit/quote), เอกสาร, โครงการที่เกี่ยวข้อง
- **Dealer Directory** (HQ): ทะเบียนดีลเลอร์ (เชียงใหม่/Master House, ระยอง/VCS Asia, เชียงราย, ตาก ฯลฯ), พื้นที่รับผิดชอบ, เป้า

### 3.2 ใบเสนอราคา / BOQ
- **Product Catalog**: EASYBUILD, RANBUILD, PREFAB + งาน Custom (โครงสร้าง/หลังคา/ฉนวน/ระบบ)
- **Master Price & Material**: ราคากลางต่อหน่วย, เกรดเหล็ก, แผ่นหลังคา, ฉนวน — คุมโดย HQ
- **BOQ Builder**: เลือกสเปก → คำนวณปริมาณเหล็ก/พื้นที่ → ออกรายการราคาอัตโนมัติ
- **Quotation**: เวอร์ชัน (rev.), ส่วนลด, ภาษี, เงื่อนไข, อนุมัติ (เกินวงเงินต้องให้ HQ อนุมัติ), ออก PDF, สถานะ
- เชื่อมกับ CRM (ลีด→ใบเสนอราคา) และ Project (ใบเสนอราคาที่ชนะ→เปิดโครงการ)

### 3.3 บริหารโครงการก่อสร้าง (ตาม 6 เฟสของ Benjamin)
แปลงใบเสนอราคาที่ชนะ → โครงการ แล้วเดินตาม workflow:

| เฟส | ระบบรองรับอะไร |
|-----|----------------|
| 1. สำรวจ & วางแผน | แบบฟอร์มสำรวจไซต์, พิกัด GPS, รูปถ่าย, ข้อจำกัดพื้นที่ |
| 2. ออกแบบ | อัปโหลดแบบ (สถาปนิก/วิศวกร), รอบรีวิว, อนุมัติแบบโดยลูกค้า |
| 3. ผลิตในโรงงาน | ติดตามสถานะการผลิตชิ้นส่วน, ล็อต, ความคืบหน้า % |
| 4. ขนส่ง & ติดตั้ง | แผนขนส่ง, ทีมหน้างาน, ความคืบหน้าการประกอบ |
| 5. ตรวจสอบ QC | เช็กลิสต์ QC, รูปก่อน/หลัง, รายการแก้ไข (punch list) |
| 6. ส่งมอบ | เอกสารส่งมอบ, คู่มือ, **การรับประกันโครงสร้าง**, ปิดงาน |

เสริม: Timeline/Gantt, Milestone, Task assignment, รายงานหน้างานประจำวัน (Daily Site Report), คลังเอกสาร & รูปภาพ, แจ้งเตือนงานล่าช้า

### 3.4 รายงาน & แดชบอร์ดผู้บริหาร
- **แดชบอร์ด HQ**: ยอดขายรวม/รายดีลเลอร์/รายภาค, Sales funnel, win rate, มูลค่า pipeline, โครงการตามสถานะ, อัตราส่งมอบตรงเวลา (on-time %)
- **แดชบอร์ดดีลเลอร์**: เป้า vs ทำได้, ลีดค้าง, ใบเสนอราคารออนุมัติ, โครงการที่ต้องลงหน้างาน
- รายงานส่งออก Excel/PDF, กรองตามช่วงเวลา/พื้นที่/สินค้า

## 4. โมดูลเสริม (Cross-cutting)
- **Auth & RBAC** — เข้าสู่ระบบ, สิทธิ์ตามบทบาท, tenant isolation
- **Master Data** — สินค้า, ราคา, วัสดุ, ดีลเลอร์, พื้นที่/จังหวัด
- **Notification** — Email + **LINE** (เหมาะกับบริบทไทย): ลีดใหม่, รออนุมัติ, งานล่าช้า, นัดหมาย
- **File/Document Storage** — แบบ, BOQ, สัญญา, รูปหน้างาน, เอกสารส่งมอบ
- **Audit Log** — ใครทำอะไรเมื่อไหร่ (สำคัญสำหรับ multi-tenant)
- **i18n** — ไทย (ค่าหลัก) / อังกฤษ

## 5. โครงสร้างข้อมูล (Data Model — ระดับ Entity)

```
Tenant/Dealer ──┐
                ├─< User >── Role
                │
Customer ──< Lead >── Activity
   │           │
   │           └──< Quotation >──< QuotationItem (BOQ line) >── Product/Material
   │                    │
   └────────────────< Project >──┬─< ProjectPhase (6) >──< Task >── Milestone
                                 ├─< Document / SitePhoto >
                                 ├─< QCChecklist >── PunchItem
                                 ├─< DailySiteReport >
                                 └─< Handover >── Warranty

Master: Product, PriceList, Material, Province/Region
System: Notification, AuditLog
```

**หลักการ:** ทุกตารางธุรกรรม (Lead, Quotation, Project, ฯลฯ) มีคอลัมน์ `dealer_id` + `created_by` เพื่อบังคับ isolation และ audit

## 7. เทคโนโลยีที่แนะนำ (Tech Stack)

| ชั้น | เทคโนโลยี | เหตุผล |
|------|-----------|--------|
| Frontend | **Next.js (React) + TypeScript + Tailwind + shadcn/ui** | เร็ว, รองรับไทย, UI สำเร็จรูป |
| Backend | Next.js API Routes / Server Actions (หรือ NestJS ถ้าทีมโต) | โค้ดเบสเดียว, deploy ง่าย |
| Database | **PostgreSQL** + Prisma ORM | รองรับ Row-Level Security / multi-tenant |
| Auth | Auth.js (NextAuth) + RBAC | ครบ, ต่อ SSO ได้ |
| Storage | S3-compatible (เช่น Cloudflare R2 / Supabase Storage) | เก็บไฟล์/รูปหน้างาน |
| Notification | **LINE Messaging API** + Email (Resend) | LINE คือช่องทางหลักในไทย |
| Dashboard | Recharts / Tremor | กราฟผู้บริหาร |
| PDF | react-pdf / Puppeteer | ใบเสนอราคา + เอกสารส่งมอบ |
| Hosting | Vercel + Managed Postgres (Neon/Supabase) | เริ่มเร็ว, สเกลได้ |

> ทางเลือก Low-code: ถ้าต้องการเร็วมากและทีมเล็ก พิจารณา **Supabase + Refine** หรือ **Directus** ได้

## 8. Roadmap การพัฒนา (แบ่งเฟส)

| เฟส | ขอบเขต | ผลลัพธ์ |
|-----|--------|---------|
| **0. Foundation** | Auth, RBAC, multi-tenant, master data, โครง UI | ล็อกอินแยก HQ/ดีลเลอร์ได้ |
| **1. CRM (MVP)** | ลีด, pipeline, ลูกค้า, ทะเบียนดีลเลอร์, รับลีดจากเว็บ | เริ่มใช้งานขายได้จริง |
| **2. Quotation/BOQ** | catalog, ราคากลาง, BOQ builder, ใบเสนอราคา PDF, อนุมัติ | ออกใบเสนอราคามาตรฐานเดียว |
| **3. Project** | 6 เฟส, task, เอกสาร, รูป, QC, ส่งมอบ, รับประกัน | ติดตามงานก่อสร้างครบวงจร |
| **4. Dashboard/Reports** | แดชบอร์ด HQ/ดีลเลอร์, รายงาน export | ผู้บริหารเห็นภาพรวม |
| **5. Integrations** | LINE OA, แจ้งเตือน, customer portal | ปิดลูปสื่อสาร |

**แนะนำเริ่ม MVP = เฟส 0 + 1** (CRM + แยกสิทธิ์ HQ/ดีลเลอร์) ให้ใช้งานจริงก่อน แล้วต่อยอด
